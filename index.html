This document provides specifications of less-rpc protocol for remote procedure call frameworks that relies on HTTP as transport protocol.

Overview #back to top

LESS-RPC is a light-weight, flexible, extendable remote procedure call (RPC) protocol that utilizes HTTP(RFC 7238) as its transport layer. It takes advantage of the HTTP headers to communicate adequate information regarding the content's format that allows server and client stub of RPC to serialize/deserialize the content. It can be seen an extension of json-rpc protocol to support any serialization format. It further defines functionalities that allows server and client stub to inform each other of supported serialization formats. It is designed to:

  • Allow utilization of different serialization format based on environment
  • Allow use of optimized serialization formats based on communicated data types

Terminology #back to top

This section will provide terminology and definitions required to interpret this document.
Protocol: the official procedure or system of rules governing affairs of state or diplomatic occasions
Stub: A class or code that provides the basic functionality for tiers involved in a remote procedure call
Server: Server is a part of a network based system that sits on a computer and listens to requests sent to a specific port number and processes them. A server application may produce a response to be sent to the client application.
Client: A part of a network based system that sends a requests to a server application.A client may or may not require a response.
It is important to notice that client and server terminology only refers to a piece of software and not a physical system since it may play the role of client and server interchangeably at any time.
Server Stub: A small application used as basis of a server RPC software
Client Stub: A small application used as basis of a client RPC software
Service: A service in LESS-RPC is specific method that will do certain processes on a set of inputs and provide the output over RPC.
Service Provider: Is a server application that implements a server stub to provide a certain functionality through RPC.

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

Any object field explained in this document followed with a "*" after the name is a required field and must be provided.

LESS-RPC Protocol #back to top

This section describes the details of less-rpc protocol including request and response messages send and received by both server and client stub applications.

HTTP - #back to top

Hypertext Transfer Protocol (HTTP) is the foundation of data communication in web and is used to exchange hypertext. LESS-RPC takes advantage of POST (RFC 7231 section 4.3.3) method to send a Service request to a Service Provider.

The POST method requests that the target resource process the representation enclosed in the request according to the resource's own specific semantics. For example, POST is used for the following functions (among others):

  • Providing a block of data, such as the fields entered into an HTML form, to a data-handling process;
  • Posting a message to a bulletin board, newsgroup, mailing list, blog, or similar group of articles;
  • Creating a new resource that has yet to be identified by the origin server; and
  • Appending data to a resource's existing representation(s).

An RPC call will fit into the first category of functionalities of a POST method which is providing data to be processed by server. Additionally, post request allows client to transfer data as content of the request that can contain plain text or binary data that enables LESS-RPC to transfer serialized format of object as either plain text (JSON, XML and etc) or binary (MsgPack, CBOR and etc.).

LESS-RPC further takes advantage of the following HTTP headers to communicate information necessary to process content of the request.

Content-type: content is design to content the encoding format of the content of an HTTP POST request. LESS-RFC follows the guidelines in RFC 6838 to transfer serialization format and version as shown below:

content-type: application/vnd.less ;lessversion=<less-rpc version> ; format=<serialization format name> ; formatversion=<serialization format version> [; parameters]

The format above allows developer to pass serialization format, version and less-rpc version that provides the minimal required information to parse and deserialize the content of POST request. serialization format name will refer to name of the method (ex. JSON, CBOR and etc.), lessversion corresponds to the version number of less-rpc being used for the content of the message(See sections Service Request and Service Response for more information). Finally, formatversion contains the version of the serialization format. All aforementioned fields are required and LESS-RPC stub will return an error (See sections) in absence of one or more of the fields.

Additionally, one can pass other parameters as described in RFC 822. As described same RFC version numbers MUST be quoted if they contain special characters ( read more).

Accept: this field is defined by HTTP to allow client pass encoding formats acceptable by client to server, LESS-RPC utilizes accept as it was intended to. The value of accept field can be set as a comma-separated list of content-type. However, it cannot be empty



accept: [content-type]+

Serialization Format - #back to top

Serialization refers to the process of encoding or translating an object in a programming language in a format that can be stored or transmitted over network. It can either be a language specific format such as Java Serialization or language independent like JSON. A list of serialization formats can be found here. Another Wikipedia article offers a comparison of existing methods.

There are numerous RPC protocols that depend on a specific serialization format including JSON-RPC, XML-RPC and etc. However, LESS-RPC offers a light-weight minimal protocol that can be used alongside any serialization format. It can further use different serialization formats for different services based on requirements and efficiency.


Service Information - #back to top

A service is a method/function that can be called by a client stub. A service is implemented and available over network by a Service Provider application through a Server Stub.

A Service is identified using a ServiceInformation object that contains the following fields:

  • name*: Name of the service. Service name should b unique for a specific service.
  • sid*: Id of service. Service Id should be unique for a specific service
A service information object can be used to request a service. An example of service information in different serialization format is shown below:

Service Provider - #back to top

A Service Provider is an application that implements a service functionality and makes it available through a sever stub. A ServiceProvider is identified using a Service Provider Information object and is required to implement certain functionalities to become fully accessible in LESS-RPC.

Service Provider Information Object
The service provider information object provides information needed to find and interpret running environment of Server Provider application. The fields for a ServiceProviderInfo object are:

  • url*: URL to be reached at.
  • port*: Port that the service provider is available at.
  • env*: Stub environment type for example Java, Python and etc.

The stub type information will be represent using the following info:

  • lang*: Language information as <name>/<version>
  • os*: Operating system information as <name>/<version>
  • compiler*: Compiler information <name>/<version>

Service Provider Information

A Service Provider application provides the following http requests:

  • /execute:

    Execute is the main Service Provider method that processes a ServiceRequest object and returns an instance of ServiceResponse object. This method can be accessed at "<url>:<port>/execute". Execute method can only be requested using a POST method. Clients should not use GET request as Service Provider application are not required to support GET method.

  • /ping:

    Ping doesn't require any parameter and returns a true or false as 1 or 0 in content. Ping can be requested using either GET or POST. It can be accessed using "<url>:<port>/ping".

  • /info:

    It can be accessed through "<url>:<port>/info" and will return a ServiceProviderInfo object. This service should be accessed through a GET request.

    Client stub should provide a list of accepted types in Accept header of Http. Client may order the accepted formats according to its preference in descending order. Server stub should choose one that is most efficient or the first one of the list that it supports to serialize the content. The chosen must be communicated to the client stub through the Content-type header of Http response. If no serialization format is provided by client or server doesn't support any of the proposed formats the server stub must return a number in text/plain format containing the proper error code (see error codes)

  • /service:

    It can be accessed through "<url>:<port>/service" and will return an instance of ServiceSupport object. "/service" request requires a ServiceInfo instance as argument and must be accessed using POST request.

    Service Support

    ServiceSupport object contains the following fields:

    • service*: Instance of ServiceInfo
    • serializers*: Array of SerializationFormat objects

    Serialization Format

    SerializationFormat object contains the following fields:

    • format*: Name of serialization format
    • version*: Version information


Service Provider must serialize message using one of the accepted serialization formats stored by client application in HTTP's Accept header. If none of the accepted serialization formats are supported by the Service Provider it must return a response containing the appropriate error message. Otherwise, the serialized message must be returned as HTTP response content and HTTP content-type header must be set by service provider following the specified format

Client Application - #back to top

Client Application may request any of the provided functions by a server application. Furthermore, when serializing the message, client application MUST provide the necessary information in "content-type" header of request packet. It MUST also provide accepted serialization formats in ACCEPT header.

Client may call "/service" to get list of supported serialization formats and serialize content of HTTP request choosing the most efficient serialization format supported. However this is not necessary though it may result in an error. When working with large data as arguments client should try to request supported serialization to avoid extra overhead unless it knows for certain what serialization is supported by the service provider.

Service Request - #back to top

Service Request

ServiceRequest object contains the following fields:

  • service*: Instance of ServiceInfo object
  • env*: StubType object instance
  • rid*: contains the request id generated for the request. request id can be used by the client to identify responses in asynchronous or batch mode.
  • args*: an array of SerializedObject instanc

Serialized Object

SerializedObject object contains the following fields:

  • clspath*: Class path for the object which depends to the client environment. It will be using class path in Java and Python but it can contain any information necessary to parse an instance of the serialized object
  • content*: Serialized data for object

Service Response - #back to top

Every call to /execute method will result with a response from the ServiceProvider which will follow the specifications below for a ServiceResponse object. LESS-RPC unlike JSON-RPC does not differentiate between normal method calls and notifications. However, the response for notifications will be minimal. Requiring a response for notification can assure client application whether the service provider has been successful in retrieving and processing the notification. ServiceResponse object contains the following fields:

  • service*: Instance of ServiceInfo object
  • rid*: contains the request id generated for the request. request id can be used by the client to identify responses in asynchronous or batch mode.
  • status*: result status code
  • content*: result object which must be contain fields of a SerializedObject if the result is a value. In case the function called is finished with an error status the content will contain additional information regarding the error in plain text format
